# AI 生成代码专用规约 (AI-Generated Code Standards)

适用于**全局**（Vibe Mode + Studio Mode 均生效）。

## 0. 立场声明 (Stance)

AI 生成的代码有其独特风险模式：**幻觉**、**过度抽象**、**上下文丢失**、**版本漂移**。
本规约独立于人类工程师规范，针对这些 AI 特有病灶。

---

## 1. 反幻觉 (Anti-Hallucination)

### 1.1 依赖存在性验证
- 每次引入新的库 / 包 / API，必须先执行验证命令：
  - Python: `pip show <pkg>` 或查询 PyPI
  - Node.js: `npm view <pkg>` 或查询 npm registry
  - Rust: `cargo search <pkg>`
- 引入前如无法验证，**必须显式标注** `// TODO: verify dependency exists`

### 1.2 API 签名核对
- 调用外部库函数前，必须查阅其官方文档确认：
  - 函数名拼写正确
  - 参数顺序、类型、可选性正确
  - 返回值类型正确
- 对不确定的 API，添加 `// ai-uncertain: <原因>` 注释

### 1.3 版本锁定
- 所有依赖必须写入依赖清单（`requirements.txt` / `package.json` / `Cargo.toml`）
- **严禁**使用 `latest` / `*` / `>=` 作为版本约束（Vibe Mode 也不行）
- 生产依赖必须锁定到具体版本或语义化范围（`^1.2.3` / `~1.2`）

---

## 2. 反过度抽象 (Anti-Over-Engineering)

### 2.1 YAGNI 原则强制
- 没有**第二个**实际调用者，禁止抽接口 / 抽象类
- 禁止为想象中的未来需求预留 hook / plugin / strategy 机制
- 禁止把一次性逻辑封装为"通用工具"

### 2.2 抽象层深度限制
- 抽象类 / 接口继承深度**不超过 2 层**
- 装饰器嵌套**不超过 3 层**
- 高阶函数嵌套**不超过 2 层**

### 2.3 命名反常识检测
- 禁止使用 `Manager`, `Handler`, `Processor`, `Helper`, `Utility` 作为核心类名
  （允许作为辅助命名，但必须有具体前缀，如 `AuthTokenManager`）
- 类名与文件名必须一致或高度相关

---

## 3. 反上下文丢失 (Anti-Context-Loss)

### 3.1 修改一致性
- 修改函数签名后，**必须全局搜索所有调用点**并同步更新
- 修改类型定义后，必须验证所有 import 位置的兼容性
- 重命名后，禁止残留旧名称的引用

### 3.2 风格一致性
- 新增代码必须匹配**同一文件**内已有代码的风格：
  - 缩进（tab vs space）
  - 引号（单 vs 双）
  - 分号（有 vs 无）
  - 命名习惯（camelCase vs snake_case）
- 跨文件风格冲突时，遵循目录内多数派

### 3.3 概念一致性
- 同一概念在项目内保持**同一命名**：
  - 不能同时存在 `userId` 和 `user_id`
  - 不能同时存在 `fetchUser` 和 `getUserData`（如无语义差异）
- 引入新概念前，先搜索是否已有等价命名

---

## 4. 可追溯性 (Traceability)

### 4.1 AI 决策标注
关键选型处添加注释，格式：
```
// ai-choice: <理由>
// ai-uncertain: <不确定点>
// ai-assumption: <未经用户确认的假设>
```

### 4.2 重构可逆性
- 大规模重构（>10 文件或 >200 行）前，必须：
  1. 在 git 中创建 checkpoint commit
  2. 或使用 worktree 隔离
- 禁止一次性重命名 + 移动 + 修改逻辑

### 4.3 生成来源标记
- 完全由 AI 生成的文件，在首部注释标注：
  ```
  // Generated by Claude Code on <日期>
  // Original prompt: <原始意图，一句话>
  ```

---

## 5. 安全强化 (Security Hardening)

### 5.1 密钥零信任
- 严禁在代码中出现任何形态的密钥，包括：
  - `"sk-..."`, `"api_key = ..."`, Base64 编码的密钥
  - 注释中的"仅供测试"密钥（即使标注"删除"）
- 一律使用环境变量 + `.env.example` 模板

### 5.2 输入信任边界
- 所有外部输入（API 参数、用户表单、文件内容、URL）必须验证
- 使用类型验证库（`pydantic` / `zod` / `joi`）
- 禁止 `any` / `unknown` 逃逸到业务逻辑层

### 5.3 依赖来源验证
- 禁止安装非官方源的包
- 谨慎对待新近发布（< 30 天）的包
- 定期运行 `npm audit` / `pip-audit` / `cargo audit`

---

## 6. 违规处理 (Enforcement)

- 检测到违规：Lead Developer 必须在 `/review` 阶段拦截
- 严重违规（密钥泄露、SQL 注入）：直接阻止 `/gate-check` 通过
- 累积违规：写入 `docs/reviews/ai-code-audit.md` 追踪
