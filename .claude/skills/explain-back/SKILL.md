# 技能：状态回顾 (/explain-back)

**执行代理**: `explainer` (解说员) 主导

---

## 概述 (Summary)

Vibe coder 经常"周末做几小时、下周再回来"——回来时脑子里的上下文早就丢了。
`/explain-back` 用非技术语言、类比修辞、可视化清单，
在 5 分钟内让用户"接上"当前项目状态，知道下一步该按哪个按钮。

**不同于 `/vibe-check`**：
- `/vibe-check`：**校准意图**（现在和最初想的一致吗？）
- `/explain-back`：**恢复上下文**（我上次做到哪了？）

---

## 触发时机 (Triggers)

- 用户直接调用 `/explain-back`
- 用户表达"我忘了做到哪了"、"当前是什么情况"、"帮我梳理下"
- 距离上次交互 > 3 天时自动建议

---

## 工作流 (Workflow)

### 阶段 1：证据收集 (Explainer 静默完成)
不打扰用户，扫描并读取：
- `.usds-mode`：当前模式
- `docs/specs/lite-spec.md` 或 `docs/specs/PRD.md`：最初意图
- `docs/specs/intent-log.md`：历史校准
- `sandbox/**/README.md` 或 `production/backlog.md`：当前工作项
- 最近 git 提交（若可用）：`git log --oneline -10`
- `docs/reviews/`：最近评审记录

### 阶段 2：状态提炼
生成三段式汇报（**全部用非技术语言**）：

#### 一、"你想做什么"
- 引用 lite-spec/PRD 的**原句**（不加工）
- 一句话总结项目目标

#### 二、"你已经做到哪"
- 上次结束时的状态（用类比说明，如"厨房已经装修好，正在挑家具"）
- 已完成的关键节点（清单，每条 1 行）
- 当前正在做的事

#### 三、"下一步该做什么"
- 3 个可选动作，标注难度（10 分钟 / 1 小时 / 半天）
- 每个动作对应一个具体的 slash command 或文件

### 阶段 3：可视化时间线
生成 ASCII 时间线（简洁，5-8 个节点）：

```
[意图] → [原型 v1] → [校准] → [原型 v2] → [审美评审] → [现在]
  ✓        ✓         ✓         ✓            ✓           ▲ 这里
```

### 阶段 4：用户交互
使用 `ask_user` 呈现 3 个下一步选项：
- [选 A：<10 分钟动作>]
- [选 B：<1 小时动作>]
- [选 C：<半天动作>]

---

## 成功门控 (Success Gate)

- 三段式汇报完整（想做什么 / 做到哪 / 下一步）
- 时间线可视化清晰（≤ 8 个节点）
- 3 个下一步选项每个都对应具体动作
- 用户从"完全没上下文"到"知道该按什么"
- **不落盘**：这是一次性汇报，不产生文档

---

## 输出格式

**这是一次直接输出到用户的技能，不落盘、不精简**。全文如下：

```markdown
# 📋 状态回顾 — <项目名>

## 你想做什么
> "<lite-spec/PRD 原句>"

一句话：<非技术语言总结>

## 你已经做到哪
你上次结束时，<类比说明>。

已完成:
- ✓ <关键节点 1>
- ✓ <关键节点 2>
- ✓ <关键节点 3>

正在做:
- ⏳ <当前工作项>

## 时间线
[意图] → [原型 v1] → [校准] → ... → [现在]
  ✓        ✓         ✓              ▲ 这里

## 下一步（选一个开始）

**A. 快速接续（10 分钟）**
<具体动作，如"跑 `/vibe-check` 校准一下方向">

**B. 认真推进（1 小时）**
<具体动作，如"打开 sandbox/xxx，修复审美评审提出的 3 个小问题">

**C. 里程碑推进（半天）**
<具体动作，如"跑 `/graduate` 把原型晋升到 src">

请选择 A / B / C，或告诉我别的方向。
```

## 特殊场景

### 项目刚起步（只有 lite-spec，没原型）
- 跳过"做到哪"章节
- 下一步只推荐 `/prototype`

### 项目卡壳（连续 3 次 `/vibe-check` 都判 pivot）
- 在汇报中主动指出"过去 X 次校准都在纠结方向"
- 下一步 A 改为"跑 `/vibe-start` 重新对齐意图"

### 有多个 sandbox 分叉存在
- 在"做到哪"中并列展示每个分叉的状态
- 下一步选项包含"跑 `/taste-review` 对比分叉"

### 已进入 Studio Mode
- 数据源从 `intent-log` 切换到 `backlog.md` 和 `RELEASE-CHECKLIST.md`
- 时间线以 sprint / milestone 为节点
- 下一步动作对齐 Studio Mode 技能（`/review` / `/gate-check`）

## 与其他技能的边界

| 关注点 | 归属 |
|---|---|
| 意图漂移分析 | `/vibe-check` |
| 上下文恢复、状态梳理 | **`/explain-back`** |
| 审美问题识别 | `/taste-review` |
| 代码质量评审 | `/review` |

---

## 输出约束

**[破例]** 本技能是**直接对用户的汇报**，输出即最终产物，不需要精简摘要。
但仍要遵守：
- 不使用未解释的技术术语
- 时间线 ≤ 8 个节点
- 下一步 ≤ 3 个选项

---

## 协作路径

- 用户选 A → 交棒对应的快速动作技能
- 用户选 B/C → 进入相应工作流
- 用户表示"我想换方向" → 建议 `/branch-vibe` 或 `/vibe-start`
